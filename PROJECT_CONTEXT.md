# DynoCore Project Context

## Назначение Документа

Этот файл должен быстро объяснять LLM и разработчикам, что представляет собой проект, какие решения уже приняты, какие технологии используются и как безопасно вносить правки.

Документ нужно обновлять:

- перед началом имплементации, если меняется идея, стек, структура или SEO-стратегия;
- после завершения крупных этапов разработки;
- после финальной сборки сайта, когда появятся реальные файлы, команды запуска, интеграции, переменные окружения и ограничения.

## Проект

`DynoCore` - SEO-ориентированный лидогенерационный сайт для услуг чип-тюнинга автомобилей в Республике Беларусь, русскоязычная выдача.

Цель сайта - получать заявки от владельцев автомобилей, которым нужна прошивка ЭБУ, Stage 1/Stage 2, диагностика, решение проблем EGR/DPF/AdBlue/SCR или консультация по безопасному приросту мощности.

Формат - многостраничный сайт, а не один лендинг. Причина: в нише есть разные коммерческие интенты, которые должны ранжироваться отдельными страницами.

## Бренд И Позиционирование

Название: `DynoCore`.

Слоган: `Точная настройка мощности. Без лишнего риска.`

Позиционирование: инженерная калибровка ЭБУ с измеримым результатом, прозрачным процессом, резервной копией прошивки, диагностикой до работ и понятными гарантийными условиями.

Коммерческий оффер: `Прошивка ЭБУ с диагностикой, расчетом прироста и резервной копией заводской программы.`

Тон коммуникации:

- профессиональный;
- уверенный, но без громких неподтвержденных обещаний;
- технологичный;
- понятный для владельца авто без глубоких технических знаний.

Не использовать агрессивный гаражный стиль, визуальный шум, огонь, шашечки, чрезмерный гоночный китч и обещания вроде `безопасно для любого авто` или `гарантированное снижение расхода`.

## Целевая Аудитория

Основные сегменты:

- владельцы дизельных и бензиновых авто, которым не хватает тяги;
- владельцы автомобилей VAG, BMW, Mercedes, Audi, Skoda, Volvo, Toyota, Lexus, Ford, Renault, Peugeot/Citroen, Geely/Belgee, Haval, Chery;
- владельцы коммерческого транспорта;
- клиенты с проблемами EGR, DPF/FAP, AdBlue/SCR, которым нужен корректный сценарий решения после диагностики;
- пользователи, сравнивающие цену, безопасность, отзывы и реальные результаты.

## SEO-Стратегия

Основной рынок: Республика Беларусь, русскоязычная выдача.

Главный подход: одна ключевая группа запросов - одна уникальная посадочная страница. Нельзя каннибализировать запросы между главной, страницами услуг, марок и городов.

Коммерческие кластеры:

- `чип тюнинг беларусь`, `чип тюнинг минск`, `чип тюнинг двигателя`;
- `прошивка эбу`, `прошивка двигателя`;
- `чип тюнинг дизель`, `чип тюнинг бензин`;
- `stage 1`, `stage 2`;
- `диагностика egr`, `проблемы dpf`, `сажевый фильтр`, `adblue scr`, а также осторожная работа с коммерческими запросами `отключение egr`, `удаление dpf`, `отключение adblue`;
- `чип тюнинг цена`, `сколько стоит чип тюнинг`;
- `чип тюнинг bmw`, `чип тюнинг audi`, `чип тюнинг volkswagen`, `чип тюнинг mercedes` и другие марки.

MVP-структура страниц:

- `/chip-tyuning/` - основная услуга;
- `/chip-tyuning-dizel/` - дизельные автомобили;
- `/chip-tyuning-benzin/` - бензиновые автомобили;
- `/stage-1/` - Stage 1;
- `/stage-2/` - Stage 2;
- `/diagnostika-egr/` - диагностика и решения по EGR;
- `/problemy-dpf-fap/` - диагностика и решения проблем DPF/FAP;
- `/diagnostika-adblue-scr/` - диагностика и решения по AdBlue/SCR;
- `/ceny/` - цены и расчет;
- `/marki/[brand]/` - страницы марок, только с уникальным контентом;

On-page требования:

- уникальные `title`, `description`, `h1` для каждой страницы;
- первый экран с услугой, регионом, преимуществом, CTA и доверием;
- FAQ на коммерческих посадочных;
- внутренние ссылки между услугами, марками, кейсами и статьями;
- schema.org JSON-LD: `Organization`, `WebSite`, `AutomotiveBusiness`/`AutoRepair`, `Service`, `FAQPage`, `BreadcrumbList`;
- `Review` и `AggregateRating` использовать только при наличии реальных отзывов на странице.

## Рекомендуемый Стек

Базовый стек:

- Framework: `Astro`;
- Language: `TypeScript`;
- Styling: `Tailwind CSS`;
- Content: `Astro Content Collections` + `MDX`;
- Interactive islands: React или Svelte только там, где нужна интерактивность;
- Validation: `Zod`;
- Forms: Astro Actions или API endpoint;
- Anti-spam: honeypot + Cloudflare Turnstile;
- Analytics: Google Analytics 4, Google Tag Manager, Яндекс Метрика;
- SEO: sitemap, robots.txt, canonical, OpenGraph, JSON-LD;
- Deploy: Cloudflare Pages, Vercel или Netlify.

Почему Astro: сайт является контентным и SEO-зависимым. Astro позволяет генерировать быстрые статические страницы, держать минимум клиентского JavaScript и удобно масштабировать посадочные страницы через content collections.

## Воронка Лида

Основной CTA: `Рассчитать прирост`.

Форма заявки должна быть короткой:

1. Марка, модель, двигатель, год.
2. Цель обращения: больше мощности, меньше расход, диагностика, консультация.
3. Контакт: телефон и предпочтительный мессенджер.
4. Опционально: VIN или фото техпаспорта, только с пояснением зачем это нужно и что поле можно пропустить.

Технические требования к заявкам:

- сохранять UTM-метки;
- отправлять событие аналитики;
- иметь honeypot/Turnstile;
- запрашивать согласие на обработку персональных данных;
- поддерживать Telegram/Viber/WhatsApp CTA;
- показывать понятное состояние успеха и ошибки;
- обещать понятный следующий шаг: расчет цены и ожидаемого прироста, обычно в течение 15 минут в рабочее время;
- не терять заявку при ошибке внешней интеграции, если будет добавлено резервное логирование.

## Новости И RSS-Агрегация

На сайте нужен блок `Последние новости в мире чип-тюнинга`. На старте используется RSS/агрегация без AI.

Цель блока:

- показать, что сайт живой и следит за индустрией;
- дать пользователю полезные новости про тюнинг, ECU, performance и новые автомобили;
- не копировать чужие полные тексты;
- отправлять пользователя на оригинальный источник по ссылке.

Рекомендуемый формат карточки:

- заголовок;
- дата;
- источник;
- краткий анонс;
- изображение, если источник передал его в RSS;
- ссылка `Читать в источнике`.

Правила использования контента:

- не публиковать полный текст чужих статей;
- хранить только краткий excerpt/summary из RSS;
- обязательно показывать источник и внешнюю ссылку;
- изображения брать только из RSS `enclosure`/`media:content` или использовать локальный fallback-баннер;
- для SEO не считать агрегированные новости основным уникальным контентом сайта.

Основные RSS-источники для реализации:

- `tuningblog.eu` - `https://www.tuningblog.eu/feed/`, крупное европейское tuning/performance медиа.
- `Viezu` - `https://viezu.com/feed/`, профильный источник про ECU remapping и tuning.
- `Fast Car` - `https://fastcar.co.uk/tuning/feed/`, tuning-гайды и car culture.
- `Motor1 Aftermarket/Tuning` - `https://motor1.com/rss/category/aftermaket-tuning/`, aftermarket/tuning новости, URL проверить при реализации.
- `autoevolution Tuning` - `https://www.autoevolution.com/rss/backend-tuning.xml`, tuning-лента, URL проверить при реализации.
- `Auto Onliner` - `https://auto.onliner.by/?feed=rss2`, локальные автомобильные новости Беларуси, показывать только релевантные авто/рынку записи.

Русскоязычные источники для мониторинга, но не как основные RSS на старте:

- `CHIPTUNER.RU` - `https://chiptuner.ru/news/` или `https://chiptuner.ru/novosti/`, RSS не подтвержден.
- `АвтоЧип` - `https://www.avtochip.ru/news/`, RSS не подтвержден.
- `Rechip` - `https://rechip.ru/news`, RSS не подтвержден.
- `SVD CHIP` - `https://svdtuning.ru/chip-tuning-news.html`, RSS не подтвержден.
- `Drive2` - использовать осторожно, RSS не подтвержден, контент пользовательский.

Технический подход:

- создать серверный fetcher RSS-лент;
- нормализовать записи к единому типу `NewsItem`;
- фильтровать дубли по URL;
- фильтровать релевантность по ключевым словам: `ECU`, `remap`, `tuning`, `chip tuning`, `diesel`, `DPF`, `EGR`, `AdBlue`, `performance`, `software`, `прошивка`, `тюнинг`, `двигатель`, `рынок авто`;
- ограничивать вывод 6-9 последними релевантными новостями;
- обновлять кэш раз в неделю через scheduled job или build hook;
- при ошибке источника не ломать сайт, а использовать последний сохраненный snapshot;
- добавить fallback-изображение в стиле `DynoCore`.

RSS-новости не должны индексироваться как отдельные SEO-страницы на старте. Это блок актуальности и доверия, а не замена экспертному блогу.

## Юридические И Этические Ограничения

Для страниц и блоков про EGR, DPF/FAP, AdBlue/SCR нельзя строить коммуникацию вокруг идеи `удалить экологию любой ценой`.

Обязательная рамка:

- сначала диагностика технического состояния автомобиля;
- приоритет исправности штатных систем для транспорта на дорогах общего пользования;
- ремонт, очистка, замена, адаптация и программные работы рассматриваются как разные сценарии;
- итоговое решение зависит от состояния автомобиля, требований эксплуатации и применимых норм;
- не обещать прохождение техосмотра, отсутствие последствий, вечное решение проблемы или одинаковый результат для всех авто.

## Контентные Принципы

Контент должен объяснять:

- что такое чип-тюнинг;
- какие преимущества и ограничения он дает;
- когда чип-тюнинг делать нельзя;
- как проходит диагностика и прошивка;
- чем отличается Stage 1 от Stage 2;
- что такое EGR, DPF/FAP, AdBlue/SCR;
- почему важна резервная копия стоковой прошивки;
- что получает клиент после заявки;
- от чего зависит цена чип-тюнинга;
- как чип-тюнинг может влиять на техосмотр, гарантию и ресурс;
- почему клиенту стоит выбрать `DynoCore`.

Нельзя давать точные или гарантированные обещания для всех авто. Формулировки должны учитывать состояние двигателя, пробег, тип мотора и качество обслуживания.

Кейсы, отзывы, графики `до/после` и рейтинги должны быть реальными. Нельзя публиковать вымышленные отзывы, псевдокейсы или `AggregateRating`, если нет подтвержденных данных на странице.

Доверительные блоки по мере готовности бизнеса:

- УНП, адрес, карта, режим работы и мессенджеры;
- фото сервиса, оборудования и процесса;
- реальные отзывы со ссылками на внешние площадки;
- гарантийные условия с исключениями;
- примеры логов и результатов до/после без завышенных обещаний.

## Env-Переменные

- `SITE_URL` - canonical URL, по умолчанию `https://dynocore.by`;
- `LEAD_WEBHOOK_URL` - опциональный server-side webhook для заявок;

- `PUBLIC_YANDEX_METRIKA_ID` - опциональная Яндекс Метрика.

## Реализованные Интеграции

- Форма заявки на React island с Zod-валидацией, согласием, honeypot, client-side UTM capture через `sessionStorage`, состояниями успеха/ошибки и событиями аналитики.
- API endpoint `src/pages/api/lead.ts`: валидирует payload, не отправляет VIN во внешний webhook напрямую, при отсутствии `LEAD_WEBHOOK_URL` пишет безопасную simulated submission в server log.
- Аналитика: placeholders для GA4, GTM и Яндекс Метрики через public env; форма отправляет submit/error/validation events, CTA/messenger/RSS clicks размечены lightweight tracking attributes.
- RSS-агрегация: `rss-parser`, список источников в `src/data/newsSources.ts`, нормализация, дедупликация, keyword-фильтр и fallback source-linked previews.
- SEO: sitemap integration, `robots.txt`, canonical, OpenGraph, JSON-LD `Organization`, `WebSite`, `AutoRepair`, `Service`, `FAQPage`, `BreadcrumbList`.

## Правила Добавления Контента

- Услуги добавлять в `src/data/services.ts`; каждая услуга должна иметь уникальные `title`, `description`, `h1`, FAQ, related links и осторожные ограничения там, где затрагиваются EGR/DPF/AdBlue.
- Марки добавлять в `src/data/brands.ts`; не копировать текст между марками без уникального блока про двигатели, риски и процесс.
- Города добавлять в `src/data/cities.ts` только при реальном обслуживании региона.
- Экспертные статьи добавлять как MDX в `src/content/blog/`; RSS-новости не превращать в индексируемые страницы.
- Кейсы и отзывы публиковать только после подтверждения реальными данными. Не добавлять `Review` или `AggregateRating` без подтвержденных отзывов на странице.

## Известные Ограничения

- Контактные данные и мессенджеры в `src/data/site.ts` являются стартовыми placeholders и должны быть заменены реальными NAP/каналами перед запуском рекламы или SEO-продвижением.
- Production POST-заявки требуют runtime. Сейчас подключен `@astrojs/node` standalone; при деплое на Vercel, Netlify или Cloudflare adapter нужно заменить на платформенный.
- RSS-источники могут быть недоступны на build-time; в этом случае показываются fallback source-linked previews.
- Реальные фото сервиса, карта, УНП, отзывы, рейтинги и подтвержденные кейсы пока не добавлены.
